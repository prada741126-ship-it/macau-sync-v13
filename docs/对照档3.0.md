### 1.3 核心設計原則

1. **Event Bus 解耦**: 模組零直接調用，透過 `Events.emit()`/`on()` 通信
2. **State Manager 單一數據源**: 所有數據透過 `State.get()`/`set()` 讀寫
3. **Store 統一入口**: 所有 localStorage 透過 `Store.save()`/`load()` 操作
4. **純函數計算層**: `calc/` 目錄全部無副作用，100% 可測試
5. **墓碑同步策略**: Firebase 刪除不設 null，寫入墓碑 `{_deleted:true}`

## 3. 核心架構

### 3.1 Event Bus (`src/core/events.js`)

**全局變量**: `Events`

| 方法 | 簽名 | 說明 |
|------|------|------|
| `on` | `(event, handler) → unsubscribeFn` | 訂閱事件，返回取消函數 |
| `off` | `(event, [handler])` | 取消訂閱；省略 handler 則取消全部 |
| `emit` | `(event, ...args)` | 發布事件，handler 錯誤不影響其他人 |
| `once` | `(event, handler)` | 一次性訂閱 |
| `debug` | `(enabled)` | 開/關調試日誌 |
| `listAll` | `() → object` | 查看所有已註冊事件及其監聽器數量 |
| `reset` | `()` | 清除所有監聽器 |

### 3.2 State Manager (`src/core/state.js`)

**全局變量**: `State`

| 方法 | 簽名 | 說明 |
|------|------|------|
| `get` | `(key) → value` | 獲取狀態值；不傳 key 返回全部 |
| `set` | `(key, value, [silent])` | 設定狀態值，自動 emit 事件 |
| `batchSet` | `(updates, [event])` | 批量設定，之後 emit 一次事件 |
| `update` | `(key, updater, [silent]) → newValue` | 不可變更新 (傳入當前值，返回新值) |
| `nextId` | `(type) → number` | 獲取自增ID (tx/fund/wallet/booking/hc) |
| `resetNextId` | `(type, value)` | 重設自增ID |
| `reset` | `()` | 重置全部狀態 |

**State 內部路徑→事件映射** (`_pathEvents`):
```
txs → TXS_LOADED          fundWithdrawals → FUND_LOADED
agentWallets → WALLETS_LOADED   agentList → AGENT_LIST_LOADED
bookings → BOOKINGS_LOADED     hotelConfig → HC_CONFIG_LOADED
workingMonth → MONTH_CHANGED   currentPage → PAGE_CHANGED
syncConnected → CONNECTION_CHANGED
```

**全局變量同步** (`window.*`):
```
window.txs, window.fundWithdrawals, window.agentWallets, window.agentList
window.workingMonth, window.nextId, window.fundNextId, window.agentWalletNextId
window.editId, window.sortState, window._syncConnected, window._draftTimer
window.__currentTimeFilter
```

**通用方法**: `save(key, data, encrypt)` / `load(key, decrypt)` / `remove(key)`

**專項存取** (每個 data type 一對 save/load):

## 6. Event Bus 事件系統

### 6.1 完整事件清單 (32個)

#### 交易事件
| 事件名 | 常量 | 參數 | 觸發時機 |
|--------|------|------|----------|
| `tx:created` | `TX_CREATED` | (tx) | createTx 完成 |
| `tx:updated` | `TX_UPDATED` | (updatedTx) | updateTx 完成 |
| `tx:deleted` | `TX_DELETED` | (deletedTx) | deleteTx 完成 |
| `txs:loaded` | `TXS_LOADED` | (txs) | State.set('txs') / 加載完成 |

#### 公基金事件
| `fund:created` | `FUND_CREATED` | (record) | createFund 完成 |
| `fund:updated` | `FUND_UPDATED` | (record) | updateFund 完成 |
| `fund:deleted` | `FUND_DELETED` | (record) | deleteFund 完成 |
| `fund:loaded` | `FUND_LOADED` | (records) | State.set('fundWithdrawals') |

#### 代理錢包事件
| `wallet:created` | `WALLET_CREATED` | ({agent, record}) | createWallet 完成 |
| `wallet:updated` | `WALLET_UPDATED` | ({agent, record}) | updateWallet 完成 |
| `wallet:deleted` | `WALLET_DELETED` | ({agent, record}) | deleteWallet 完成 |
| `wallets:loaded` | `WALLETS_LOADED` | (wallets) | State.set('agentWallets') |

#### 代理名單事件
| `agentList:updated` | `AGENT_LIST_UPDATED` | (list) | CRUD 完成 |
| `agentList:loaded`  | `AGENT_LIST_LOADED`  | (list) | State.set('agentList') |

#### 訂房事件
| `booking:created` | `BOOKING_CREATED` | (booking) | createBooking |
| `booking:updated` | `BOOKING_UPDATED` | (booking) | updateBooking |
| `booking:deleted` | `BOOKING_DELETED` | (booking) | deleteBooking |
| `bookings:loaded` | `BOOKINGS_LOADED` | (bookings) | State.set('bookings') |

### 8.2 同步步驟詳解

#### A. 初始化流程 (`initFirebase`)

```
1. 立即嘗試 _doInitFirebase()
2. 若失敗 → load 事件重試 (兼容 document.readyState===complete)
3. 若仍失敗 → 1秒輪詢 × 30次 (檢測 typeof firebase !== 'undefined')
4. 成功後 → _onFirebaseReady()
   ├── startWatchers()     ← 啟動9個即時監聽器
   └── syncDownloadAll()   ← 首次拉取遠端數據
5. _watchConnection() 連通後 2秒 → syncUploadAll() 安全網
```

#### B. CRUD 即時寫入 (`firebase.js`)

每個 CRUD 操作完成後，立即調用對應的同步函數：

| CRUD 操作 | 同步函數 | 墓碑函數 |
|-----------|---------|---------|
| createTx | `syncTxToFirebase(tx)` | — |
| updateTx | `syncTxToFirebase(updated)` | — |
| deleteTx | `trackRecentlyDeleted('tx', fbKey)` | `removeTxFromFirebase(fbKey)` |
| createFund | `syncFundToFirebase(record)` | — |
| updateFund | `syncFundToFirebase(updated)` | — |
| deleteFund | `trackRecentlyDeleted('fund', fbKey)` | `removeFundFromFirebase(fbKey)` |
| createWallet | `syncWalletToFirebase(agent, record)` | — |
| updateWallet | `syncWalletToFirebase(agent, updated)` | — |
| deleteWallet | `trackRecentlyDeleted('wallet', fbKey)` | `removeWalletFromFirebase(agent, fbKey)` |
| createBooking | `syncBookingToFirebase(booking)` | — |
| updateBooking | `syncBookingToFirebase(updated)` | — |
| deleteBooking | `trackRecentlyDeleted('booking', fbKey)` | `removeBookingFromFirebase(fbKey)` |
| createHC | `syncHCConfigToFirebase(entry)` | — |

| renameAgent | `syncAgentListToFirebase(list)` + tx sync + wallet sync | wallet tombstone |

#### C. 墓碑策略 (Tombstone Strategy)

```
刪除時不設 null，寫入墓碑對象: { _fbKey, _deleted: true, _updatedAt }

流程:
1. deleteTx(fbKey) → splice 本地數組
2. trackRecentlyDeleted('tx', fbKey) ← 60秒追蹤
3. removeTxFromFirebase(fbKey) → 寫墓碑到 Firebase
4. 另一端 watcher 收到 → mergeTxs 中墓碑 _updatedAt > 本地 → 墓碑勝出
5. 客戶端過濾: merged.filter(r => !r._deleted) → 移除墓碑
6. Firebase 保留墓碑，直到所有設備同步完畢

syncUploadAll 保護:
- transaction 中檢查 isRecentlyDeleted('tx', fbKey)
- 若為最近刪除 → 強制寫入墓碑（防止遠端舊數據"復活"）
```

#### D. 即時監聽器 (9個 watchers)

| # | 路徑 | 合併策略 | 墓碑過濾 |
|---|------|---------|---------|
| 1 | `TXS` | `mergeTxs(local, remote)` | 客戶端 filter `!_deleted` |
| 2 | `FUND` | `mergeTxs(local, remote)` | 客戶端 filter `!_deleted` |
| 3 | `AGENT_LIST` | 智能合併 (區分刪除/新增/CASE3) | `isRecentlyDeleted('agent', name)` 排除 |
| 4 | `AGENT_WALLETS` | `mergeWallets(local, remote)` | 客戶端 filter `!_deleted` |
| 5 | `WORKING_MONTH` | 直接覆蓋 | N/A |
| 6 | `RM_BOOKINGS` | `mergeBookings(local, remote)` | 客戶端 filter `!_deleted` |
| 7 | `ARCHIVES` | 直接覆蓋 | N/A |
| 8 | `HC_CONFIG` | `mergeTxs(local, remote)` | 客戶端 filter `!_deleted` |
| 9 | `CLEARED_AT` | 清除檢測 | 新增清除事件 → 清空本地 |

#### E. 批量上傳 (`syncUploadAll`)

```
8 路徑 transaction 原子合併:
1. TXS       → transaction(mergeTxs(local, remote)) + recentlyDeleted 墓碑安全網
2. FUND      → transaction(mergeTxs + tombstone 安全網)
3. AGENT_LIST→ transaction(本地為空跳過 / 本地權威)
4. WALLETS   → transaction(mergeWallets + tombstone 安全網)
5. WORKING_MONTH → set()
6. RM_BOOKINGS → transaction(mergeBookings + tombstone 安全網)
7. HC_CONFIG → transaction(mergeTxs + tombstone 安全網)
8. ARCHIVES  → set()
```

#### F. 上傳隊列 (`uploader.js`)

```
enqueueUpload(task):
  → 推入 _uploadQueue
  → _processQueue() 串行處理
  → _db 為 null 時延遲 1s 重試 (不退隊)
  → 每任務間隔 100ms (避免 Firebase 限制)
```

### 8.3 衝突合併策略

| 函數 | 策略 | 適用數據 |
|------|------|----------|
| `mergeTxs(local, remote)` | 時間戳決勝: `_updatedAt` 大者勝出 | TXS, FUND, HC_CONFIG |
| `mergeWallets(local, remote)` | 時間戳決勝 (巢狀結構) | AGENT_WALLETS |
| `mergeBookings(local, remote)` | 時間戳決勝 (與 mergeTxs 一致) | RM_BOOKINGS |
| `mergeAgentLists(local, remote)` | 雙向並集合併 + 排序 | AGENT_LIST |
| `mergeArrays(local, remote)` | 本地優先 + 墓碑過濾 | 通用數組 |

**關鍵規則**: mergeTxs/mergeWallets/mergeBookings 返回**含墓碑**結果 → watchers.js 客戶端側 filter `!_deleted`

## 14. 修復手冊

### 14.1 同步問題診斷步驟

#### A. 確認 Firebase 連接狀態

1. 開啟 `?debug=1` 參數查看控制台日誌
2. 檢查日誌中是否有 `✅ Connected! _db ready`
3. 若為 `❌ Firebase SDK failed to load after 30s`:
   - 檢查是否可訪問 `cdn.jsdelivr.net`
   - 清除瀏覽器緩存後重試

#### B. 確認數據寫入

1. 檢查 `🚀 syncTx WRITE:` / `✅ syncTx OK:` 日誌
2. 若為 `syncTx deferred (_db null) → auto-enqueue`:
   - 等待 Firebase SDK 載入
   - 上傳隊列會自動重試
3. 若為 `❌ syncTx FAILED:`:
   - 檢查 Firebase 配額/權限
   - 網絡是否穩定

#### C. 確認數據接收

1. 檢查 `[v13:watchers] TXS CHANGED: X → Y entries` 日誌
2. 若無 CHANGED 日誌:
   - 確認兩端使用相同 Firebase 項目
   - 檢查 FB_PATH 路徑是否一致
   - 確認數據已寫入 Firebase RTDB

#### D. 確認刪除同步

1. 檢查 `🗑️ removeTx TOMBSTONE:` / `✅ removeTx TOMBSTONE OK:` 日誌
2. 若刪除後數據「復活」:
   - 檢查墓碑是否成功寫入 Firebase
   - `mergeTxs` 中時間戳是否正確
   - `syncUploadAll` 是否觸發了 recentlyDeleted 安全網

### 14.2 常見問題與修復

| 問題 | 症狀 | 檢查點 | 修復 |
|------|------|--------|------|
| CDN 加載失敗 | `Firebase SDK failed to load after 30s` | cdn.jsdelivr.net 可訪問性 | 清除緩存；切換網絡 |
| 數據不寫入 | 所有 `sync* WRITE` 不出現 | `_db` 是否為 null | 確認 Firebase 已初始化 |
| 數據不接收 | 無 `CHANGED` 日誌 | watcher 是否啟動 | 檢查 `startWatchers` 調用 |
| 刪除復活 | 刪後數據重新出現 | 墓碑是否寫入 | 檢查 `remove*FromFirebase` 調用 |
| 版本緩存 | 手機加載舊代碼 | app.js?v 參數 | `APP.VERSION` 包含時間戳 |
| 代理名單不同步 | addAgent 後手機沒反應 | `syncAgentListToFirebase` | 檢查 Firebase AGENT_LIST 路徑 |
| 錢包餘額錯誤 | calcAgentBalance 返回異常 | txs/agentWallets 數據一致性 | 調用 `validateMonthBalance` |

### 14.3 構建與部署流程

```bash
# 1. 修改 src/ 或 css/ 源文件
# 2. 構建 + 測試
node build.js
# 3. 確認 60/60 PASS
# 4. 複製到 Pages repo
cp dist/* ../macau-sync-v13-pages/
# 5. 提交並推送
cd ../macau-sync-v13-pages
git add -A
git commit -m "fix: 描述"
git push
```

**⚠️ 重要規則**:
- 永遠修改 `src/` 或 `css/`，不直接改 `dist/` 或 `docs/`
- 測試不通過不允許部署
- HTML onclick 必須在 bridge.js 提供對應全局函數
- 硬編碼顏色通過 UI_COLORS 常量引用
- try-catch 必須有 console.error 輸出
- Pages 讀取的是 `macau-sync-v13-pages` repo

### 14.5 同步架構全貌速查

```
merger.js       → mergeTxs/mergeBookings/mergeWallets 均返回含墓碑結果
watchers.js     → 合併後客戶端過濾墓碑，State 中不含墓碑
uploader.js     → transaction 寫回含墓碑數據，Firebase 中保留墓碑
syncDownloadAll → 同 watchers，客戶端過濾墓碑

墓碑生命週期:
  刪除時寫入 Firebase → 各端監聽器收到後本地過濾
  → 墓碑保留在 Firebase 直至所有端同步
```
