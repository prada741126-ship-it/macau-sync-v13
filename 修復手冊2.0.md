## 問題一：代理錢包刪除不同步

### 🔍 症狀描述
- 手機上刪除代理錢包紀錄，電腦端沒有同步
- 或電腦端刪除，手機端沒有同步

### 🔍 根本原因
v10.x 舊版 `deleteAgentWallet()` 只寫 localStorage，沒有調用 Firebase 寫入。
v13 架構下，`data/wallets.js` 的 `deleteWallet()` 必須調用 `removeWalletFromFirebase()`。

### 🔧 診斷步驟（v13）

**步驟 1：檢查 Firebase 上的墓碑對象**
```javascript
// 在瀏覽器 Console 執行
firebase.database().ref('macau_data/agentWallets/代理名稱').once('value')
  .then(s => console.log('Firebase 資料:', s.val()));
// 刪除後應看到 { _fbKey: '...', _deleted: true, _updatedAt: ... }
```

**步驟 2：檢查 deleteWallet 是否有調用 removeWalletFromFirebase**
在 `data/wallets.js` 的 `deleteWallet()` 結尾確認：
```javascript
// ✅ 正確：刪除後寫墓碑到 Firebase
removeWalletFromFirebase(agent, fbKey);
```

**步驟 3：檢查 watcher 是否過濾墓碑**
在 `sync/watchers.js` 的 AGENT_WALLETS watcher 中確認：
```javascript
// ✅ 正確：mergeWallets 後客戶端側過濾墓碑
var merged = mergeWallets(local, remote);
// 過濾每個代理的墓碑記錄
for (var agent in merged) {
  merged[agent] = merged[agent].filter(function(r) { return !r._deleted; });
}
```

### 🔧 解決方案（v13 架構）

**v13 正確流程**：
```
1. data/wallets.js: deleteWallet(agent, fbKey)
   → State.update → Store.saveWallets
   → removeWalletFromFirebase(agent, fbKey)  // 寫墓碑到 Firebase
   → Events.emit(WALLET_DELETED)

2. sync/firebase.js: removeWalletFromFirebase(agent, fbKey)
   → db.ref('macau_data/agentWallets/' + agent + '/' + fbKey)
     .set({ _fbKey, _deleted: true, _updatedAt: Date.now() })

3. sync/watchers.js: AGENT_WALLETS watcher
   → 收到 onValue → mergeWallets(local, remote)
   → 客戶端過濾 _deleted → State.set('agentWallets', filtered)
   → Events.emit(WALLETS_LOADED) → UI 刷新
```

### ✅ 驗證方法
1. 在手機上刪除一筆代理錢包紀錄
2. 在電腦 Console 看 `[Sync ↓] agentWallets CHANGED` 日誌
3. 確認電腦上的該筆紀錄也消失
4. 檢查 Firebase Console 是否有墓碑對象（`_deleted: true`）

## 問題二：最後一筆記錄無法刪除

### 🔍 症狀描述
- 刪除代理的最後一筆錢包紀錄後，頁面重新整理又會回來
- 或刪除後 Firebase listener 又把舊數據拉回來

### 🔍 根本原因（v13 架構）

**原因 A：mergeWallets 過早過濾墓碑**
`merger.js` 的 `mergeWallets()` 如果在合併階段就過濾 `_deleted: true`，會導致墓碑不被寫回 Firebase，其他設備永久收不到刪除通知。

**原因 B：watcher 收到 null 時未清空本地**
當某代理的所有錢包記錄都被刪除時，Firebase 上該代理節點為 `null`，watcher 必須正確處理。

### 🔧 解決方案（v13 架構）

**原則：墓碑不過濾於 merger.js，過濾於 watchers.js**

```javascript
// ✅ 正確：mergeWallets (merger.js) — 不過濾墓碑
function mergeWallets(local, remote) {
  var allAgents = {};
  var localAgents = Object.keys(local || {});
  var remoteAgents = Object.keys(remote || {});
  // 收集所有代理
  for (var i = 0; i < localAgents.length; i++) allAgents[localAgents[i]] = true;
  for (var i = 0; i < remoteAgents.length; i++) allAgents[remoteAgents[i]] = true;

  var result = {};
  var agents = Object.keys(allAgents);
  for (var i = 0; i < agents.length; i++) {
    var agent = agents[i];
    result[agent] = mergeTxs(local[agent] || [], remote[agent] || []);
    // ★ 不過濾墓碑 — 墓碑需要寫回 Firebase
  }
  return result;
}

// ✅ 正確：watchers.js — 客戶端側過濾墓碑
var merged = mergeWallets(localWallets, remoteWallets);
var filtered = {};
var agents = Object.keys(merged);
for (var i = 0; i < agents.length; i++) {
  var agent = agents[i];
  filtered[agent] = merged[agent].filter(function(r) {
    return !r._deleted;
  });
  // 空的代理也要保留（允許空陣列）
}
State.set('agentWallets', filtered);
```

### ✅ 驗證方法
1. 新增一筆代理錢包紀錄
2. 刪除該筆紀錄（此時代理的錢包應該是空的）
3. 重新整理頁面 → 確認該筆紀錄沒有再出現
4. 確認 Firebase 上的該代理節點下只有墓碑對象

## 問題三：版本更新後客戶端看不到新版本

### 🔍 症狀描述
- 部署新版本到 GitHub Pages，但電腦和手機都還是顯示舊版本
- 即使重新整理頁面也沒用

### 🔍 根本原因（v13 架構）

**原因 A：GitHub Pages CDN 緩存 app.js**
`app.js?v=13.0.0` 版本號永不變 → CDN 認定同一文件 → 直接返回緩存。
這是 v13 期間最致命的發現之一（第十輪修復）。

**原因 B：Service Worker (SW) 頑固快取（v10.x 舊版）**
v13 架構已移除 Service Worker，但若用戶瀏覽器仍有舊 SW 註冊，會繼續攔截請求。

### 🔧 解決方案（v13 架構）

**方案 A：動態版本號（v13 推薦）**

`build.js` 將 `__VERSION__` 替換為 `版本號.時間戳`：
```javascript
// build.js
var cacheVersion = version + '.' + Date.now();  // v13.0.2.1781262147266
content = content.replace(/__VERSION__/g, cacheVersion);
```

template.html：
```html
<script src="app.js?v=__VERSION__"></script>
<!-- 部署後變為: <script src="app.js?v=13.0.2.1781262147266"></script> -->
<!-- 每次部署 URL 不同 → CDN 強制刷新 -->
```

**方案 B：版本快取清除（v13 已有）**

`app.js` 的 `checkVersionCache()` 在啟動時檢測：
```javascript
function checkVersionCache() {
  var saved = Store.loadAppVersion();
  var current = APP.VERSION;
  if (saved && saved !== current) {
    // 版本變更 → 強制刷新
    location.replace(location.href.split('?')[0] + '?_v=' + Date.now());
  }
  Store.saveAppVersion(current);
}
```

**方案 C：移除 Service Worker（v13 已執行）**
- v13 的 template.html 不再註冊 Service Worker
- 在 `<head>` 最前加入 SW 自殺腳本（參見 v1.0 手冊）

### ✅ 驗證方法
1. 部署新版本
2. 在客戶端 F12 → Network → 看 app.js 請求 URL 是否含新時間戳
3. 看 Console 有無 `Version changed, reloading...` 日誌
4. 確認頁面底部版本號為新版本

## 問題五：交易登錄後資料不同步

### 🔍 症狀描述
- 手機上新增交易紀錄，電腦端沒有同步

### 🔍 根本原因（v13 架構）

**原因 A：CRUD 函數從不調用 Firebase 寫入（v13 第四輪修復）**
`syncTxToFirebase` / `syncFundToFirebase` / `syncWalletToFirebase` / `syncBookingToFirebase`

這個 Bug 從 v13 第一天就存在。

**原因 B：_db 為 null 時 sync 函數靜默跳過**
所有 sync 函數第一行 `if (!_db) return` → Firebase 未初始化時靜默丟棄寫入。

### 🔧 解決方案（v13 架構）

**修正後的 data/transactions.js createTx：**
```javascript
function createTx(formData) {
  // ... 計算金額 ...
  var tx = { /* ... */ _createdAt: Date.now(), _updatedAt: Date.now() };

  State.update('txs', function(txs) { txs.push(tx); return txs; });
  Store.saveTxs(State.get('txs'));

  // ★ 立即同步到 Firebase
  syncTxToFirebase(tx);

  Events.emit(EVENTS.TX_CREATED, tx);
  return tx;
}
```

**修正後的 syncTxToFirebase（自動入隊）：**
```javascript
function syncTxToFirebase(tx) {
  if (!tx._fbKey) tx._fbKey = generateFbKey();
  var db = getDB();
  if (!db) {
    // ★ _db 為 null → 入隊等待（不丟失寫入）
    enqueueUpload(function() {
      syncTxToFirebase(tx);  // 重試時重新讀取最新 tx
    });
    return;
  }
  db.ref(FB_PATH.TXS + '/' + tx._fbKey).set(tx, function(err) {
    if (err) {
      console.error('❌ syncTx FAILED:', err);
      enqueueUpload(function() { syncTxToFirebase(tx); });
    } else {
      console.log('✅ syncTx OK:', tx._fbKey);
    }
  });
}
```

## 問題六：交易刪除後又跑回來

### 🔍 症狀描述
- 刪除交易紀錄後，頁面重新整理又會跑回來

### 🔍 根本原因（v13 架構）

**墓碑機制未正確實現**：
- v13 第十一輪修復前，`removeTxFromFirebase` 使用 `.set(null)` → key 直接消失
- `mergeTxs` 是並集策略 → 本地有 key、遠端沒有 → key 被保留 → 刪除失效

### 🔧 解決方案（v13 墓碑機制）

**刪除時寫墓碑（firebase.js）：**
```javascript
function removeTxFromFirebase(fbKey) {
  var db = getDB();
  if (!db) {
    enqueueUpload(function() { removeTxFromFirebase(fbKey); });
    return;
  }
  // ★ 寫墓碑對象，不設 null
  var tombstone = {
    _fbKey: fbKey,
    _deleted: true,
    _updatedAt: Date.now()
  };
  db.ref(FB_PATH.TXS + '/' + fbKey).set(tombstone, function(err) {
    if (err) {
      console.error('❌ removeTx FAILED:', err);
      enqueueUpload(function() { removeTxFromFirebase(fbKey); });
    } else {
      console.log('✅ removeTx OK:', fbKey);
    }
  });
}
```

**合併時保留墓碑（merger.js）：**
```javascript
function mergeTxs(local, remote) {
  var map = {};
  // local 全部入 map
  for (var i = 0; i < local.length; i++) {
    var tx = local[i];
    if (tx._fbKey) map[tx._fbKey] = tx;
  }
  // remote 逐項合併
  for (var i = 0; i < remote.length; i++) {
    var rx = remote[i];
    if (!rx._fbKey) continue;
    var existing = map[rx._fbKey];
    if (!existing || (rx._updatedAt || 0) >= (existing._updatedAt || 0)) {
      map[rx._fbKey] = rx;  // ★ 墓碑也保留
    }
  }
  // ★ 不過濾墓碑 — 由 watchers.js 過濾
  var result = [];
  var keys = Object.keys(map);
  for (var i = 0; i < keys.length; i++) result.push(map[keys[i]]);
  return result;
}
```

**客戶端過濾墓碑（watchers.js）：**
```javascript
var merged = mergeTxs(localTxs, remoteTxs);
var filtered = merged.filter(function(tx) { return !tx._deleted; });
State.set('txs', filtered);
```

1. 新增一筆交易
2. 刪除該筆紀錄
3. 重新整理頁面 → 確認該筆紀錄沒有再出現

## 問題七：交易計算錯誤

### 🔍 症狀描述
- 佣金（comm）計算錯誤
- 碼糧（bonus）計算錯誤
- 公基金（fund）計算錯誤

### 🔍 根本原因（v13 架構）

v13 通過 `calc/finance.js` 純函數層解決，100% 可測試。

**關鍵公式**：
- `calcComm(vol, rate)` → `Math.ceil(vol × 10000 × rate / 100)`
- `calcFund(comm, bonus)` → `comm - bonus`
- `calcUndrawn(bonus, drawn)` → `Math.max(0, bonus - drawn)`

### 🔧 解決方案（v13 架構）

v13 已通過 50 項自動測試確保計算公式正確。任何計算 Bug 應在 `calc/finance.js` 中修復，
並在 `test/test.js` 中添加對應測試案例。


```
  T2: DOMContentLoaded → initFirebase() 被調用
```

這是 v13 第五輪修復的核心發現。

### 🔧 診斷步驟

```javascript

console.log('typeof firebase:', typeof firebase);  // "undefined" 或 "object"
// 若 firebase 是 object 但 _db 是 null → 確認是本問題
```

### 🔧 解決方案（v13 已實施）

**`initFirebase()` 加入重試機制（firebase.js）：**
```javascript
function initFirebase() {
  // 嘗試 1：立即初始化
  if (_doInitFirebase()) return;

  // 嘗試 2：load 事件重試
  if (document.readyState === 'complete') {
    // 頁面已加載完成 → addEventListener('load') 是 no-op → setTimeout
    setTimeout(function() {
      if (_doInitFirebase()) return;
    }, 500);
  } else {
    window.addEventListener('load', function() {
      _doInitFirebase();
    });
  }

  // 嘗試 3：輪詢兜底（1s × 30次 = 最多 30 秒）
  var attempts = 0;
  var pollInterval = setInterval(function() {
    attempts++;
    if (_doInitFirebase() || attempts >= 30) {
      clearInterval(pollInterval);
      if (attempts >= 30 && !getDB()) {
        console.error('❌ Firebase 初始化失敗（30次輪詢後仍未就緒）');
      }
    }
  }, 1000);
}

function _doInitFirebase() {
  if (getDB()) return true;  // 已初始化
  if (typeof firebase === 'undefined') return false;

  try {
    firebase.initializeApp(FIREBASE_CONFIG);
    _db = firebase.database();
    _watchConnection();
    return true;
  } catch(e) {
    console.error('Firebase init error:', e);
    return false;
  }
}
```

### 🔧 解決方案（v13 智能策略）

**代理名單同步使用「remote 權威覆蓋」，絕不取并集。**

```javascript
// AGENT_LIST watcher — 智能策略
var _agentListFirst = true;  // 首次觸發標記

function onAgentListValue(snapshot) {
  var remote = snapshot.val() || [];
  var local = State.get('agentList') || [];

  // CASE 0: 首次觸發 → 跳過（避免頁面載入時覆蓋本地）
  if (_agentListFirst) {
    _agentListFirst = false;
    // 例外：本地為空 + 遠端有數據 → 接受遠端
    if (local.length === 0 && remote.length > 0) {
      State.set('agentList', remote);
      Store.saveAgentList(remote);
      Events.emit(EVENTS.AGENT_LIST_UPDATED, remote);
    }
    return;
  }

  // CASE 1: remote=[] && local>0 → 刪除同步（清空本地）
  if (remote.length === 0 && local.length > 0) {
    State.set('agentList', []);
    Store.saveAgentList([]);
    Events.emit(EVENTS.AGENT_LIST_UPDATED, []);
    return;
  }

  // CASE 2: local=[] && remote>0 → 新增同步（接受遠端）
  if (local.length === 0 && remote.length > 0) {
    State.set('agentList', remote);
    Store.saveAgentList(remote);
    Events.emit(EVENTS.AGENT_LIST_UPDATED, remote);
    return;
  }

  // CASE 3: 兩邊都有數據 → remote 權威覆蓋
  var localSorted = local.slice().sort().join(',');
  var remoteSorted = remote.slice().sort().join(',');
  if (localSorted !== remoteSorted) {
    State.set('agentList', remote);
    Store.saveAgentList(remote);
    Events.emit(EVENTS.AGENT_LIST_UPDATED, remote);
  }
  // ★ 絕不取并集 — 取并集 = 被刪代理復活
}
```

```
mergeTxs (merger.js):
  合併 local + remote → 保留墓碑 (_deleted: true)
  ★ 不過濾墓碑 — 墓碑需要寫回 Firebase 供其他設備接收

watchers.js (客戶端):
  mergeTxs 結果 → filter(r => !r._deleted)
  ★ State 中不含墓碑，墓碑僅存在於 Firebase

uploader.js (批量推送):
  transaction() → mergeTxs(local, remote)
  ★ 寫回 Firebase 時含墓碑，確保其他設備能收到刪除通知
```

**墓碑生命週期圖：**
```
Firebase:    [墓碑存在於 RTDB，包含 _deleted:true]
                  ↓ onValue
watchers.js: mergeTxs(含墓碑) → 過濾墓碑 → State(無墓碑)
                  ↓ Events.emit
UI:          正確移除該記錄
                  ↓ syncUploadAll
uploader.js: mergeTxs(含墓碑) → transaction 寫回 Firebase(含墓碑)
                  ↓
其他設備:    watchers 收到墓碑 → 過濾 → State 移除 → UI 更新
```

**未來可加定期清理機制**：檢查墓碑的 `_updatedAt` > 30 天 → 可以安全刪除（所有設備應已同步）。

### ✅ 驗證方法
1. 設備 A 刪除一筆交易 → 確認 Firebase 有墓碑對象
2. 設備 B watcher 接收 → 確認該筆交易消失
3. 設備 B 點擊同步 → 確認墓碑仍在 Firebase（不被清除）

## 問題十八：首次連通缺下行同步

### 🔍 症狀描述
- Firebase 首次連通後，只執行上傳（syncUploadAll），不執行下載
- 新設備首次打開時，無法從 Firebase 拉取已有數據
- 設備 A 有數據，設備 B 首次連通後數據為空

### 🔍 根本原因

`_watchConnection()` 的首次連通回調中只調用了 `syncUploadAll()`（上傳），
沒有 `syncDownloadAll()`（下載）。這意味著：
- 設備 B 首次連通 → 推送自己的空數據 → 但從不拉取設備 A 的數據

### 🔧 解決方案（v13 已實施）

**首次連通時雙向同步：**
```javascript
function _watchConnection() {
  var connectedRef = _db.ref('.info/connected');
  var prevConnected = false;
  var initialSyncDone = false;

  connectedRef.on('value', function(snap) {
    var connected = snap.val() === true;

    if (connected && !initialSyncDone) {
      initialSyncDone = true;
      // ★ 雙向同步：先下拉遠端數據，再推送本地數據
      setTimeout(function() {
        syncDownloadAll(function() {
          console.log('[Sync] 首次下載完成');
          syncUploadAll(function() {
            console.log('[Sync] 首次上傳完成');
          });
        });
      }, 2000);
    }

    if (!connected && prevConnected) {
      console.warn('[Sync] Firebase 連線中斷');
    }

    prevConnected = connected;
    State.set('syncConnected', connected);
  });
}
```

### ✅ 驗證方法
1. 在設備 A 新增 3 筆交易
2. 設備 B 首次打開 → 確認看到設備 A 的 3 筆交易
3. Console 應看到 `[Sync] 首次下載完成`

## 問題十九：mergeTxs 過濾墓碑的設計陷阱

### 🔍 症狀描述
- 修復刪除同步後，新增了墓碑機制，但刪除仍然不同步
- 在同一次修復中，改了 merger.js 又改了 watchers.js，互相矛盾

### 🔍 根本原因

**mergeTxs 在合併階段過濾墓碑** → 墓碑從 Firebase 被清除 → 其他設備永遠收不到刪除通知。

這是 v13 第十二~十三輪修復之間的核心矛盾：
- 第十一輪：mergeTxs 加墓碑過濾（錯誤！）
- 第十三輪：發現 uploader.js 的 transaction 把過濾後的結果（無墓碑）寫回 Firebase → 墓碑被清除
- 最終修復：mergeTxs **不過濾**墓碑，由 watchers.js 在客戶端側過濾

### 🔧 解決方案（v13 最終設計）

**核心原則：三個層次各司其職**

| 層次 | 文件 | 職責 |
|------|------|------|
| **merger.js** | mergeTxs/mergeWallets/mergeBookings | 合併 local+remote，保留墓碑，**不過濾** |
| **watchers.js** | 即時監聽器 | 合併後**客戶端過濾**墓碑，State 中無墓碑 |
| **uploader.js** | syncUploadAll transaction | 寫回 Firebase 時**含墓碑**，確保其他設備能接收 |

```
merger.js:     mergeTxs 返回含墓碑結果（不過濾）
                    ↓
watchers.js:   合併 → filter(!_deleted) → State.set（無墓碑）
                    ↓
uploader.js:   transaction → mergeTxs(含墓碑) → 寫回 Firebase（含墓碑）
```

### ✅ 驗證方法
1. 檢查 `merger.js` 的 mergeTxs：不應有 `.filter(r => !r._deleted)`
2. 檢查 `watchers.js` 的所有 watcher：mergeTxs 結果後應有墓碑過濾
3. 檢查 `uploader.js` 的 syncUploadAll：transaction 結果不應過濾墓碑

```javascript
function debugLog(className, msg) {
```

## 緊急處理 SOP（v13 架構）

### 如果客戶回報「刪除後又跑回來」

1. **立即確認問題範圍**
   - 哪種數據？（交易/公基金/代理錢包/訂房）
   - 只有特定設備？還是所有設備？

2. **檢查 Firebase 墓碑**
   ```javascript
   firebase.database().ref('macau_data/txs').once('value')
     .then(s => {
       var d = s.val();
       var keys = Object.keys(d);
       var tombstones = keys.filter(k => d[k]._deleted);
       console.log('墓碑數量:', tombstones.length, '/', keys.length);
     });
   ```

3. **檢查 merger.js — 是否過早過濾墓碑**
   - 確認 mergeTxs 沒有 `.filter(r => !r._deleted)`

4. **檢查 watchers.js — 是否正確過濾墓碑**
   - 確認合併後有 `.filter(r => !r._deleted)`

5. **臨時解決方案**
   - 請客戶清除瀏覽器快取 + 硬刷新（Ctrl+Shift+R）
   - 在 Firebase Console 手動檢查並刪除異常數據
   - 使用 `?debug=1` 參數查看手機端日誌

6. **長期解決方案**
   - 修復後 `node build.js` → 確認 50/50 PASS
   - `bash deploy.sh` 部署
   - 通知所有使用者硬刷新頁面

### 如果客戶回報「新增後不同步」

1. **檢查 Firebase 是否連通**
   ```javascript
   console.log('_db:', _db);
   ```

2. **檢查 write callback**
   - Console 應看到 `✅ syncTx OK`
   - 若無 → 檢查 data/ CRUD 函數是否調用了 sync*ToFirebase

3. **檢查 CDN**
   - Console 應有 `Firebase initialized successfully`
   - 若無 → CDN 被牆或載入失敗

4. **檢查 watcher**
   - Console 應看到 `[Sync ↓] TXS CHANGED`
   - 若無 → watchers.js 的 onValue 未生效
